# PR Comment Investigation Guide

コメント調査時の具体的な手法リファレンス。SKILL.md のワークフロー Step 3 で参照する。

## 1. コード影響範囲分析

### 対象コードの読み込み
```
Read: コメント対象ファイルを読み込み、対象行の前後50行のコンテキストを把握
```

### 呼び出し元・依存先の追跡
```
LSP goToDefinition: 対象コードが参照している型・関数の定義元を特定
LSP findReferences: 対象コード（関数・メソッド・変数）がどこから参照されているかを特定
LSP incomingCalls: 対象関数を呼び出している関数を特定
LSP outgoingCalls: 対象関数が呼び出している関数を特定
```

### 同一パターンの使用箇所検索
```
Grep: 対象の関数名・パターン名でコードベース全体を検索
Glob: 同じパッケージ・ディレクトリ内の関連ファイルを特定
```

## 2. テスト調査

### 関連テストの特定
```
Glob: "*_test.go" で対象パッケージのテストファイルを検索
Grep: 対象関数名をテストファイル内で検索
```

### テスト実行
```bash
# 対象パッケージのテスト実行
go test ./path/to/package/ -v -run TestName

# カバレッジ付き実行
go test ./path/to/package/ -v -cover -run TestName
```

### テストカバレッジ確認
- 対象コードがテストでカバーされているかを確認
- カバーされていない場合、対応時にテスト追加が必要かを判断材料にする

## 3. 既存パターン整合性

### コーディングパターンの確認
```
Grep: 同じパッケージ内の類似処理を検索して、既存のコーディングパターンを把握
Read: AGENTS.md / ガイドラインファイルを確認して規約との整合性をチェック
```

### 確認ポイント
- エラーハンドリングパターン（wrap方式、カスタムエラー型の使用）
- ログ出力パターン（ログレベル、フォーマット）
- バリデーションパターン
- DB操作パターン（トランザクション管理、クエリ構造）
- レスポンス構造パターン

## 4. パフォーマンス影響評価

### チェック項目
- N+1クエリの可能性
- 不要なメモリ確保（大きなスライスの事前確保等）
- ループ内のDB/API呼び出し
- 不要なデータの読み込み（SELECT * 等）
- goroutine リークの可能性

## 5. 対応推奨度の判定基準

| 推奨度 | 基準 |
|--------|------|
| **Must** | バグ修正、セキュリティ問題、データ不整合を引き起こす問題 |
| **Should** | コード品質改善、パフォーマンス改善、テスト追加、ガイドライン違反の修正 |
| **Could** | リファクタリング提案、コードスタイル改善、より良い命名 |
| **Skip** | 好みの問題、既存パターンとの整合性があるもの、PRスコープ外の改善 |
